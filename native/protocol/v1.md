# IPC Protocol v1

This document explains the contract in prose. The versioned machine-readable source of truth for the replatform foundation lives in [v1.contract.json](v1.contract.json), with generated artifacts emitted under [generated/](generated/). After contract edits, run `npm run protocol:generate`; before trusting a change, run `npm run protocol:check`.

## Transport

- shell starts the engine as a supervised child process
- shell writes UTF-8 JSON lines to engine `stdin`
- engine writes UTF-8 JSON lines to `stdout`
- engine logs diagnostic text to `stderr`

Every transport message must be serialized as one compact JSON object per line.

- `engine.ping` currently accepts no parameters and returns:
  - `protocol`
  - `engineVersion`

## Message Types

### Request

```json
{ "type": "request", "id": "1", "method": "engine.ping", "params": {} }
```

### Response

Successful response:

```json
{ "type": "response", "id": "1", "ok": true, "result": { "engineVersion": "0.1.0" } }
```

Error response:

```json
{ "type": "response", "id": "1", "ok": false, "error": { "code": "UNKNOWN_METHOD", "message": "Unsupported method" } }
```

### Event

```json
{ "type": "event", "event": "engine.ready", "payload": { "protocol": "1", "engineVersion": "0.1.0" } }
```

Structured startup failure event:

```json
{
  "type": "event",
  "event": "engine.startupFailed",
  "payload": {
    "stage": "bootstrap",
    "code": "BOOTSTRAP_FAILED",
    "message": "...",
    "paths": { "logFilePath": "/logs/engine.log" }
  }
}
```

## Startup Handshake

1. engine emits `engine.ready`
2. shell sends `health.snapshot`
3. shell sends `app.snapshot`
4. shell decides whether the startup target is operator UI or commissioning from `app.snapshot`
5. shell may request non-gating domain snapshots such as `commissioning.snapshot`, `lighting.snapshot`, `audio.snapshot`, and `planning.snapshot`
6. shell opens the operator surface only after both startup-gating snapshots succeed
7. shell fails startup if any startup step does not complete inside its watchdog window

In the current scaffold the shell still renders the runtime scaffold immediately, but the protocol and shell state machine now model this startup order explicitly.

## Dev-Only Operator Verification

The parity program uses one additive dev-only IPC method for deterministic live verification:

- `dev.parityFixture.load`

This method is not part of the user-facing product contract. It exists so the native shell can launch into engine-owned parity states and be checked in the real app, not only in fixture-level harnesses.

`dev.parityFixture.load` currently accepts:

- `fixtureId`: required, one of the current fixture ids below

Successful responses return the loaded fixture metadata and emit `app.changed`, `commissioning.changed`, and `planning.changed`.

Current fixture ids:

- `planning-empty`
- `planning-populated`
- `lighting-populated`
- `audio-populated`
- `setup-required`
- `setup-ready`

## Native Control-Surface Bridge

The native runtime now also exposes an engine-owned local HTTP bridge for Bitfocus Companion and related control-surface tooling. This bridge is separate from the JSON-lines IPC transport above.

- default base URL: `http://127.0.0.1:38201`
- override port with `SSE_CONTROL_SURFACE_PORT`
- current bridge endpoints:
  - `GET /api/deck/context`
  - `GET /api/deck/lcd?key=...`
  - `POST /api/deck/action`
  - `POST /api/deck/light-action`
  - `POST /api/deck/audio-action`
- the generated Companion profile now targets this native bridge instead of the Electron web server

## Startup Lifecycle States

The shell startup state machine currently uses these lifecycle states:

- `idle`
- `launching-process`
- `waiting-for-ready-event`
- `waiting-for-health-snapshot`
- `waiting-for-app-snapshot`
- `ready`
- `failed`

Contract rules:

- `engine.ready` means the engine process is alive, storage bootstrap completed, and the protocol version is compatible enough to continue startup.
- `health.snapshot` provides operational readiness and diagnostics metadata.
- `app.snapshot` provides engine-owned shell state, UI-routing state, and startup gating data such as workspace/window restoration, commissioning status, and startup target surface.
- `app.changed` is a non-startup domain event used to tell the shell that app-owned routing or commissioning state mutated and snapshots should be refreshed.
- `commissioning.changed` is a non-startup domain event used to tell the shell that commissioning probe/config/seed state mutated and snapshots should be refreshed.
- `planning.changed` is a non-startup domain event used to tell the shell that planning state mutated and snapshots should be refreshed.
- `lighting.changed` is a non-startup domain event used to tell the shell that lighting state mutated and snapshots should be refreshed.
- `audio.changed` is a non-startup domain event used to tell the shell that audio state mutated and snapshots should be refreshed.
- `settings.changed` is a non-startup domain event used to tell the shell that persisted shell settings mutated outside a direct `settings.update` request and the authoritative `app.snapshot` should be refreshed.
- `support.changed` is a non-startup domain event used to tell the shell that backup/export state mutated and support snapshots should be refreshed.
- `ready` means the shell has received both health and app snapshots, not merely that the process exists.

## Rules

- `id` is required for requests and responses
- `event` is required for events
- all payloads must be JSON objects
- unknown methods return structured errors, not crashes
- protocol version mismatch should surface as `engine.ready` incompatibility or a structured startup error
- shell provides runtime directories through environment variables before launch:
  - `SSE_APP_DATA_DIR`
  - `SSE_LOG_DIR`
  - `SSE_PROTOCOL_VERSION`
  - optional `SSE_UPDATE_REPOSITORY_PATH`
- `health.snapshot` should expose storage metadata needed for startup diagnostics:
  - schema version
  - journal mode

## Ownership

- The shell owns process supervision, retry behavior, and desktop presentation.
- The engine owns persisted application state, commissioning state, and UI-routing decisions derived from persisted state.
- React does not issue raw protocol messages directly. All transport calls go through shell-side adapter methods.
- The shell may cache snapshot data for rendering, but the engine remains the source of truth.

## Versioning

- Additive changes inside existing result objects are allowed in protocol v1 if old clients can ignore the new fields safely.
- Removing fields, changing field types, or changing startup sequencing requires a protocol version bump.
- A new startup-critical method required for shell boot must be documented in the startup handshake and treated as part of the versioned contract.

## Error Taxonomy

Structured errors and startup failures should use stable codes grouped by responsibility:

- startup:
  - `BOOTSTRAP_FAILED`
  - `STARTUP_TIMEOUT`
  - `HEALTH_SNAPSHOT_FAILED`
  - `APP_SNAPSHOT_FAILED`
- process supervision:
  - `PROCESS_ERROR`
  - `PROCESS_EXIT`
  - `ENGINE_BINARY_MISSING`
  - `ENGINE_NOT_RUNNING`
- protocol:
  - `UNKNOWN_METHOD`
  - `INVALID_PARAMS`
  - `INVALID_ENGINE_MESSAGE`
  - `PROTOCOL_MISMATCH`
- persistence:
  - `STORAGE_ERROR`
  - `RUNTIME_DIRECTORY_ERROR`
- import:
  - `IMPORT_SOURCE_NOT_FOUND`
  - `IMPORT_FAILED`
  - `IMPORT_REQUIRES_FORCE`
- lighting:
  - `LIGHTING_UNCONFIGURED`
  - `LIGHTING_NOT_VERIFIED`
  - `LIGHTING_PROBE_FAILED`
  - `LIGHTING_SCENE_NOT_FOUND`
  - `LIGHTING_SCENE_RECALL_FAILED`
- audio:
  - `AUDIO_TRANSPORT_UNAVAILABLE`
  - `AUDIO_NOT_VERIFIED`
  - `AUDIO_PROBE_FAILED`
  - `AUDIO_SYNC_FAILED`
  - `AUDIO_SNAPSHOT_NOT_FOUND`
  - `AUDIO_SNAPSHOT_RECALL_FAILED`

Codes may appear either in engine error responses or shell-generated failure surfaces. The meaning of a code must remain stable once released.

## Design Notes

- JSON Lines is chosen for v1 because it is easy to debug and cross-platform over a supervised child-process boundary
- if binary framing is needed later, preserve the message schema and swap only the framing layer
- the shell should treat `engine.ready` as "process is alive and protocol-compatible", not as "system is fully healthy"
- the engine should bootstrap storage paths before emitting `engine.ready`, so first-launch storage failures are caught before the shell proceeds
- `app.snapshot` is the first non-shell UI-driving contract and exists specifically to keep commissioning/dashboard routing out of ad hoc shell state
