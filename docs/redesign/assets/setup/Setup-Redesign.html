Three directions for the
Setup workspace.

Setup is a modal mode (dashboard header hidden). Two operator mental modes share it: Commissioning — first-run, infrequent, sequential; and Support — incident response, occasional, triage-shaped. Directions differ on three axes: how aggressively to reframe Commissioning as a wizard, how to express Support's incident-order IA, and how to lay out SetupControlSurfacePanel — the ~1300-LOC dominant surface that maps Bitfocus Companion pages to Stream Deck+ buttons and encoders.

Direction A · Retune

Keep the two-column panel shape. Fix what the audit called out — kill the hero stack, collapse the rail into an accordion, honor incident-order in Support, go full-width.

The conservative option. Every panel and every label that exists today still exists. What changes: the 120 px eyebrow/title/description stack becomes a 44 px title bar; the 4-card left rail becomes a workflow-aware accordion (active step expanded, others are compact header rows); the fake "Workflow: Import first" KPI is killed; the 1720 letterbox opens to full 2560; and Support reorders to put Restore first because that's incident order. No new navigation metaphor. Least risk. Ships fastest.

Direction B · Runner

Commissioning becomes a true wizard. One step at a time, centered and narrative. Support is a corner toggle because it's incident-only.

The reframe. Accepts that commissioning is a sequential task (import → probe → map → verify → publish) and builds a step runner around it. The four equal cards disappear: each step is a full-focus surface with its own layout and its own CTA. Progress is the hero header: ①──②──●──④──⑤. Support is downplayed to a small diagnostics chip in the corner — clicking it swaps the entire surface to a Support dashboard that is itself full-width and triage-ordered. The letterbox is now a deliberate narrative device, not a mistake.

C
Direction C · Inspector

Setup is a live device inspector. Tree on the left, detail on the right. Same shape whether you're commissioning a controller or triaging a failed deployment.

The bold option. Treats the whole Setup surface as devtools for the studio. A persistent entity tree on the left (profile → pages → buttons[] · encoders[]) holds everything; selecting any node fills the right pane with its binding detail, last-probed timestamp, and any errors. Commissioning isn't a wizard — it's "create and fill out nodes". Support is the same tree rooted at backups / diagnostics / paths. The physical 4×2 grid is a small always-visible inspector inside the tree; the detail pane is where real work happens. Dense. Reference-tool energy. Best for power users and for the specific audit finding that commissioning feels sequential but actually lets you jump around.

Δ
ConsoleTheme deltas · proposed

New tokens and Console* variants this redesign would add to the theme.

All in keeping with dark-only, sage-as-healthy, mono-for-data. Nothing breaks existing callers. Justified per the brief's "call out name + value + rationale" rule.

--bg-inspector
#0d0e15
Between bg-1 and bg-2. The detail pane in Direction C needs to read as "one level deeper" than a normal panel without looking like bg-2 (which is a chip surface). If we only ship A or B, omit this.
--bg-step-active
#0d100c · sage-tinted
Active wizard step card background in Direction B. Matches brand-mark background so the current-step affordance reads as "this is where your attention should be."
--step-indicator
sage · width:72px
Reserved width per step token in the wizard progress bar (Dir B). Not a color — a layout primitive. Makes the header bar consistent regardless of step count.
ConsoleStepHeader
variant
New primitive: step index · label · status dot · optional estimate. Used in Dir B wizard and reused as a list row in Dir A's collapsed accordion. Replaces the existing hero+KPI block everywhere.
ConsoleEntityTree
variant (Dir C only)
Indented tree with mono labels, keyboard-nav, inline status dots. If we don't pick C, delete this from the spec. Too much new surface area otherwise.
ConsoleSurfaceGrid
variant
Physical-layout renderer for Stream Deck+: 4×2 button grid + 4 encoder row + touch strip. Used by all three directions. Replaces the button/dial mini-nav in SetupControlSurfacePanel.
